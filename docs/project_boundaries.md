# Project Boundaries

ROCm-Oxide owns the core Rust-first AMD GPU SDK/runtime/toolchain. It should
stay focused on building, loading, launching, diagnosing, and validating GPU
workloads on ROCm rather than growing into an application framework or
simulation product.

## Core SDK Ownership

ROCm-Oxide owns:

- HIP, HIPRTC, COMGR, and ROCm toolchain integration.
- Rust-authored device-kernel authoring and compilation flow.
- `.hsaco` code object generation.
- Kernel metadata and manifest generation.
- Generated typed host bindings for compiled device kernels.
- Host runtime APIs for devices, modules, kernels, launches, and diagnostics.
- `DeviceBuffer`, `ManagedBuffer`, `PinnedHostBuffer`, `Stream`, `Event`, and
  graph helper APIs.
- `GpuArray`, `GpuArray2D`, and focused GPU helper APIs that make SDK examples
  easier to write.
- `cargo rocm-oxide` diagnostics, build, verification, pipeline, profiling, and
  project-generation tooling.
- Release gates, doctor checks, consumer-smoke checks, and documentation for the
  SDK surface.

The core crate and supporting tools should avoid app-specific assumptions. When
a feature is needed only by one visual demo or experiment, it belongs in that
demo unless it is clearly a reusable SDK primitive.

## Out Of Scope

ROCm-Oxide does not own:

- Robotics simulation.
- Full scene editors.
- Physics engines.
- ROS2 bridges.
- Dataset management systems.
- Isaac-Sim-like application UX.
- CUDA binary compatibility or `libcuda.so` emulation.
- Multi-backend application frameworks.

Those can be downstream projects that depend on ROCm-Oxide. They should not
copy private internals or force simulator-specific APIs into the core SDK.

## Examples And Demo Projects

`examples/` contains small SDK examples, diagnostics, and release probes. These
examples should teach or validate one ROCm-Oxide concept at a time:

- HIPRTC runtime compilation.
- Rust-authored device kernels.
- Generated typed bindings.
- Runtime memory, stream, event, graph, and launch APIs.
- Focused `rocm_oxide::gpu` helper usage.
- Validation and profiling probes.

`demo-projects/` contains larger downstream-style applications built with
ROCm-Oxide. These projects may include windowing, Vulkan presentation, capture,
image output, benchmark harnesses, or richer controls. They are useful as
integration demos, but they are not part of the core SDK stability contract.

## Downstream Repositories

Downstream repositories such as `rocm-oxide-sim` and
`rocm-oxide-robotics-lab` should consume ROCm-Oxide as a dependency. During the
preview phase that usually means one of:

- A local scaffold generated by `cargo rocm-oxide new`, with relative `path`
  dependencies back to a ROCm-Oxide checkout.
- A git scaffold generated with `cargo rocm-oxide new --git`, plus an installed
  `cargo-rocm-oxide` wrapper for generation/checking and `rocm-oxide-build`
  binary on `PATH` or through `ROCM_OXIDE_BUILD` for builds.

Downstream projects should keep application state, simulation models, scene UX,
datasets, and robotics integrations outside this repository. If they discover a
missing reusable SDK primitive, that primitive can be proposed here with a small
example and a verification path.
