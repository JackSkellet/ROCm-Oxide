# Scaffold required files

Every consumer project generated by `cargo rocm-oxide new` contains three
files that are **required for the build to work**. Omitting any one of them
produces an opaque error that does not point back to the scaffold.

> These files are generated automatically. Do not create them by hand.
> If they are missing, re-run `cargo rocm-oxide new <path>` to regenerate
> the scaffold, or run `cargo rocm-oxide check-consumer` to diagnose
> what is absent.

---

## Required files

| File | Purpose | Missing symptom |
|---|---|---|
| `build.rs` | Invokes `rocm-oxide-build` to compile the Rust GPU kernel; sets `ROCM_OXIDE_DEVICE_BINDINGS`, `ROCM_OXIDE_DEVICE_HSACO`, `ROCM_OXIDE_DEVICE_METADATA`, and `ROCM_OXIDE_DEVICE_MANIFEST` env vars that `src/main.rs` reads at compile time | `error: environment variable 'ROCM_OXIDE_DEVICE_BINDINGS' not defined` at compile time |
| `rust-toolchain.toml` | Pins the same nightly Rust channel used by the ROCm-Oxide workspace; requests `rust-src`, `clippy`, and `rustfmt` components | Cargo uses stable Rust; `cargo build` fails with `error[E0635]: unknown feature \`build-std\`` or similar `-Z` flag rejection |
| `device-spike/Cargo.toml` | Declares the GPU device crate with the correct `rocm-oxide-device` and `rocm-oxide-kernel` path dependencies | Cargo resolver error citing a missing or wrong `Cargo.toml` |

---

## `build.rs`

`build.rs` is the glue between the host build and the GPU kernel compiler.
It:

1. Runs `rocm-oxide-build` (found via `ROCM_OXIDE_BUILD` env var, a sibling
   binary on `PATH`, or `cargo run` against the source workspace).
2. Copies the generated `.hsaco`, `.bindings.rs`, `.metadata.json`, and
   `.manifest.json` into `OUT_DIR`.
3. Emits four `cargo:rustc-env` lines so that `src/main.rs` can access the
   artifacts via `env!("ROCM_OXIDE_DEVICE_BINDINGS")` etc.

If `build.rs` is absent the four env vars are never set and every `env!()`
call in `src/main.rs` fails at compile time with an "environment variable not
defined" error that does not explain the root cause.

---

## `rust-toolchain.toml`

The GPU kernel compilation step (`-Z build-std=core --target
amdgcn-amd-amdhsa`) requires a nightly Rust toolchain with `rust-src`
installed. Without `rust-toolchain.toml`, `cargo` picks the system default
(usually stable) and rejects the `-Z` flags outright.

The generated `rust-toolchain.toml` looks like:

```toml
[toolchain]
channel = "nightly"
components = ["rust-src", "clippy", "rustfmt"]
```

The exact nightly pin is inherited from the ROCm-Oxide source workspace's own
`rust-toolchain.toml` at scaffold generation time.

## Diagnosing a broken consumer project

Run the following from inside the consumer project directory:

```sh
cargo rocm-oxide check-consumer
```

This validates each path dependency, checks that `build.rs` is present and
sets `ROCM_OXIDE_DEVICE_BINDINGS`, and checks that `rust-toolchain.toml`
requests `rust-src`. Output mirrors the `doctor` PASS/FAIL style.
